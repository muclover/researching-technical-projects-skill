CUDA-OXIDE
02 · 定位

它解决的是 Rust device kernel，而不是 Rust host binding。

cudarc 让 Rust 调 CUDA Driver API；cuda-oxide 试图让 Rust 代码经过 rustc、MIR、Pliron、LLVM NVPTX 生成 PTX。

读法

把它看成编译器和 runtime 入口，而不是语法糖。

判断

当 Rust 工程化、安全边界和 CUDA 模型同时重要时，它才有差异。

03 · 增量

Rust 的价值来自类型约束、工程统一和 unsafe 边界。

ThreadIndex + DisjointSlice 把“每线程写不同槽位”做成 API 约束；Cargo 统一 host/device；高级硬件能力留给显式 unsafe。

读法

把它看成编译器和 runtime 入口，而不是语法糖。

判断

04 · 架构

项目的护城河在编译链，而不是单个 crate。

proc macro 标记 kernel；rustc backend 收集 device call graph；Stable MIR 进入 Pliron；LLVM NVPTX 输出 PTX。

06 · 竞品

Triton 和 TileLang 仍更适合快速写 AI 算子。

matmul、attention、dequant、pipeline、autotune 更贴近 DSL/tile-level 框架；cuda-oxide 更适合 Rust-first + CUDA SIMT 完整控制。

所以：评估 cuda-oxide 时，重点看目标 workload、调试链路、unsafe 审查和工具链复现，而不是只看 hello kernel。

07 · 风险

最大的风险不是语法，而是工具链和 soundness。

错误可能来自 rustc、macro、MIR lowering、Pliron、LLVM IR、llc、PTX 和 CUDA runtime；`index_2d(row_stride)` 已记录 soundness gap。

所以：评估 cuda-oxide 时，重点看目标 workload、调试链路、unsafe 审查和工具链复现，而不是只看 hello kernel。

08 · 厂商启示

NPU 厂商应建设语言入口，而不是照抄 CUDA 语法。

短期优先 Rust host runtime、driver binding、算子辅助库、IR pass、debug tooling 和安全 kernel API。

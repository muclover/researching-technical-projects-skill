生产前必须验证的问题

cuda-oxide 的价值不在于立刻替代 CUDA C++，而在于验证 Rust 能否进入 CUDA SIMT kernel。建议技术战略和 GPU kernel 团队做小规模验证，生产系统团队继续使用成熟主路径。

为什么值得看

它不是 CUDA Driver binding，而是一条从 rustc、Stable MIR、Pliron、LLVM NVPTX 到 PTX 的编译链。

  • Rust 类型系统可约束线程索引、输出写入和 unsafe 边界。
  • host/device 留在 Cargo 体系，减少 FFI 和构建系统断层。
  • TMA、WGMMA、cluster、Blackwell tensor core 已进入能力矩阵。

为什么不能急

项目仍是 early alpha，真正风险集中在工具链稳定性和工程可复现性。

  • 依赖 nightly、rustc_private、Stable MIR、LLVM 21+ 和 CUDA Toolkit。
  • `index_2d(row_stride)` 已记录 soundness gap。
  • GPU 示例依赖硬件环境，公开 CI 不能证明全量持续通过。

推荐推进路线从观察到 PoC，再到生产前评估

阶段 1
环境复现
验证 `doctor/run/pipeline`、driver、CUDA Toolkit、LLVM 和 nightly 组合。
阶段 2
非关键 kernel
迁移 vecadd、map、reduction 或一个业务辅助 kernel，记录 unsafe 范围。
阶段 3
生产门槛
比较 CUDA C++/Triton baseline、调试成本、CI、容器化和 license 风险。
对 NPU/AI 加速器厂商，短期不要复制 CUDA 语法。更实际的切入点是 Rust host runtime、安全 driver binding、IR pass、debug tooling 和安全 kernel API。
